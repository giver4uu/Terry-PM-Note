## 1. 시작하기

### 앱 실행

```bash
cd ontology-manager-app
npm install   # 처음만
npm run dev   # 개발 서버 시작
```

브라우저에서 `http://localhost:5173` 접속

### 첫 방문 시

처음 접속하면 **4단계 온보딩 튜토리얼**이 자동으로 표시됩니다.
- 이후에도 헤더의 **? 버튼**을 클릭하면 다시 볼 수 있습니다.

## 2. 인터페이스 이해하기

### 화면 구성

```
┌─────────────────────────────────────────────────────────────┐
│  [로고] 온톨로지 매니저    [검증배지] [Simulator] [...] [?] │  ← 헤더
├─────────────────────────────────────────────────────────────┤
│                                              │ [⚙️] │      │
│                                              │ [✓]  │ 우측 │
│              캔버스 영역                      │      │ 패널 │
│         (클래스 노드 & 관계)                  │      │      │
│                                              │      │      │
└─────────────────────────────────────────────────────────────┘
```

### 헤더 버튼

| 버튼 | 기능 |
|------|------|
| **검증 배지** | 에러/경고 개수 표시 (클릭 불가) |
| **Simulator** | 쿼리 시뮬레이터 열기 |
| **🌙/☀️** | 다크/라이트 모드 전환 |
| **🌐** | 언어 변경 (English/한국어) |
| **Export JSON** | 온톨로지를 JSON 파일로 저장 |
| **Markdown** | 온톨로지를 문서로 내보내기 |
| **?** | 온보딩 튜토리얼 다시 보기 |

### 우측 패널 토글

캔버스 오른쪽에 두 개의 아이콘 버튼이 있습니다:
- **⚙️ (Settings)**: 선택한 클래스의 프로퍼티 편집
- **✓ (Validation)**: 검증 결과 확인

## 3. 클래스 관리

### 새 클래스 추가

1. 캔버스 우측 상단의 **"+ Add Class"** 버튼 클릭
2. 또는 빈 공간에 드래그 앤 드롭

### 클래스 편집

1. 캔버스에서 **클래스 노드 클릭** (선택됨)
2. 우측 패널에서 **⚙️ Properties** 탭 선택
3. 편집 가능 항목:
   - **클래스 이름**: 상단에서 직접 수정
   - **설명**: Description 필드
   - **프로퍼티**: 이름, 타입, 필수 여부, 설명

### 프로퍼티 타입

| 타입 | 용도 | 예시 |
|------|------|------|
| `text` | 문자열 | 이름, 이메일, 상태 |
| `number` | 숫자 | 점수, 연봉, 기간 |
| `date` | 날짜/시간 | 지원일, 면접일 |
| `boolean` | 참/거짓 | 승인 여부, 필수 여부 |

### 클래스 삭제

- 클래스 선택 후 우측 패널에서 **삭제 버튼** 클릭
- 또는 키보드 **Delete/Backspace** 키

## 4. 관계 설정

### 관계 생성

1. **출발 클래스**의 가장자리(handle)에서 드래그 시작
2. **도착 클래스**의 가장자리에서 드롭
3. 새 관계가 생성됩니다 (기본: "RELATION")

### 관계 편집

1. 캔버스에서 **엣지(화살표) 클릭**
2. 우측 패널에서 편집:
   - **관계명**: 예) `APPLIES_TO`, `HAS`, `CREATES`
   - **Cardinality**: `1:1`, `1:N`, `N:1`, `N:M`
   - **설명**: 관계의 의미 설명

### Cardinality 가이드

| 표기 | 의미 | 예시 |
|------|------|------|
| `1:1` | 1대1 | Candidate ↔ Application (한 지원서는 한 후보자) |
| `1:N` | 1대다 | Recruiter → Applications (한 담당자가 여러 지원 관리) |
| `N:1` | 다대1 | Applications → Job Posting (여러 지원이 한 공고로) |
| `N:M` | 다대다 | Interviewers ↔ Interviews (면접관-면접 다대다) |

## 5. 검증 기능

### 자동 검증

- 노드나 엣지를 수정하면 **1초 후 자동 검증**됩니다
- 헤더의 **검증 배지**에서 결과 확인

### 검증 패널 열기

1. 우측 토글 버튼에서 **✓ (Validation)** 클릭
2. 에러/경고/정보 개수와 상세 목록 확인
3. **이슈 클릭 → 해당 노드로 자동 이동**

### 검증 항목 (5개)

| 검증기 | 수준 | 설명 |
|--------|------|------|
| **DuplicateClassValidator** | 🔴 에러 | 같은 이름의 클래스 중복 |
| **CircularReferenceValidator** | 🔴 에러 | A→B→C→A 순환 참조 |
| **RequiredPropertyValidator** | 🟡 경고 | ATS 권장 프로퍼티 누락 |
| **CardinalityConsistencyValidator** | 🔴 에러 | Cardinality 위반 |
| **OrphanNodeValidator** | 🟡 경고 | 연결 없는 고립 클래스 |

### 에러 vs 경고

- **🔴 에러**: 반드시 수정 필요 (온톨로지 일관성 문제)
- **🟡 경고**: 권장사항 (무시 가능하나 개선 권장)

## 6. 내보내기

### JSON 내보내기

헤더의 **"Export JSON"** 버튼 클릭
- 파일명: `ontology-export-{timestamp}.json`
- 용도: 백업, 버전 관리, 개발팀 공유

### Markdown 내보내기

헤더의 **"Markdown"** 버튼 (초록색) 클릭
- 파일명: `ontology-{timestamp}.md`
- 내용: 클래스 목록, 프로퍼티 테이블, 관계 표
- 용도: **개발팀과 공유**, 문서화, PRD 첨부

## 7. 시뮬레이터

헤더의 **"Simulator"** 버튼 클릭

### 기능

자연어 질문을 입력하면 현재 온톨로지로 해당 쿼리가 가능한지 검증합니다.

### 사용 예시

```
입력: "지원자의 면접 평가 점수를 보여줘"

결과:
✅ 쿼리 가능
경로: Candidate → Application → Interview → Evaluation
사용 프로퍼티: Candidate.name, Evaluation.score
```

### 활용

- 온톨로지 설계 검증
- 누락된 클래스/프로퍼티 발견
- 개발팀에 쿼리 요구사항 전달

## 8. 팁 & 베스트 프랙티스

### 클래스 명명 규칙

✅ **좋은 예**: `Candidate`, `JobPosting`, `Interview`  
❌ **나쁜 예**: `candidate_info`, `job-posting`, `인터뷰`

- **PascalCase** 사용
- 영문 권장 (개발팀 호환)
- 단수형 사용 (Candidates ❌ → Candidate ✅)

### 프로퍼티 명명 규칙

✅ **좋은 예**: `applied_date`, `email`, `overall_score`  
❌ **나쁜 예**: `AppliedDate`, `이메일`, `score-overall`

- **snake_case** 사용
- 영문 소문자
- 의미가 명확하게

### 관계 명명 규칙

✅ **좋은 예**: `APPLIES_TO`, `HAS`, `CREATES`, `BELONGS_TO`  
❌ **나쁜 예**: `appliesTo`, `has-relation`

- **SCREAMING_SNAKE_CASE** 사용
- 동사 사용
- 방향이 명확하게 (source가 주어)

### 검증 체크리스트

온톨로지 완성 전 확인할 사항:

- [ ] 에러 0개 (헤더 배지 확인)
- [ ] 고립 클래스 없음
- [ ] 모든 클래스에 최소 1개 이상 프로퍼티
- [ ] 주요 클래스에 description 작성
- [ ] Markdown 내보내기 후 리뷰
