# 📘 PM용 가이드

## 질문 설계를 위한 Intent / Cognitive 설계 원칙

### 이 문서의 목적

이 문서는 PM이 다음을 **직접 설계·조정**할 수 있도록 돕기 위한 것이다.

- 어떤 질문을 시스템에 허용할지
- 그 질문이 어떤 **의도(intent)** 와 **관점(cognitive)** 을 갖는지
- AI가 안정적으로 이해할 수 있도록 **의미적 제약을 어디까지 둘지**

👉 목표는 **자유로운 질문 설계**이지,

👉 **AI 로직을 PM이 직접 만드는 것**은 아니다.

---

## 1. 기본 원칙 요약 (가장 중요)

> PM은 ‘질문의 의미 구조’를 설계한다.
수식, 데이터 구조, 실행 방식은 시스템이 책임진다.
> 

그래서 PM이 만지는 영역은 딱 여기까지다:

- ✅ Intent 정의 / 선택
- ✅ Cognitive 정의 / 조합
- ❌ Metric 계산식
- ❌ SQL / 데이터 구조
- ❌ 실행 단계

---

## 2. Intent 설계 가이드 (PM 관점)

### 2.1 Intent란 무엇인가 (PM 관점 정의)

> “이 질문을 통해 사용자가 궁극적으로 얻고 싶은 것”
> 

PM은 질문을 만들 때 항상 이 질문을 스스로에게 해야 한다.

> “이 질문의 목적은 무엇인가?”
> 

---

### 2.2 PM이 쓸 수 있는 Intent (기본 세트)

아래 7개는 **시스템 공통 intent**이며,

PM은 **이 중에서만 선택**한다.

| Intent | PM 관점 설명 | 이런 질문 |
| --- | --- | --- |
| analysis | 현황·패턴을 보고 싶다 | “요즘 채용 상황 어때?” |
| diagnosis | 문제 원인을 알고 싶다 | “왜 탈락률이 높아?” |
| comparison | 차이를 비교하고 싶다 | “이번 달 vs 지난달” |
| forecast | 미래를 예측하고 싶다 | “다음 분기 채용은?” |
| recommendation | 뭘 해야 할지 알고 싶다 | “어떻게 개선할까?” |
| lookup | 숫자/리스트만 보고 싶다 | “현재 몇 명이야?” |
| explanation | 결과 해석이 필요하다 | “이 결과 무슨 뜻이야?” |

---

### 2.3 PM이 Intent를 바꿀 수 있는 범위

### ✅ 허용

- 질문별로 intent 직접 지정
- 동일 질문을 다른 intent로 재정의
    
    (예: analysis → diagnosis)
    

### ❌ 금지

- 새로운 intent 추가
- intent를 2개 이상 동시에 지정

👉 **Intent는 반드시 1개**

---

## 3. Cognitive 설계 가이드 (PM 관점)

### 3.1 Cognitive란 무엇인가 (PM 관점 정의)

> “이 질문을 어떤 관점·방식으로 보고 싶은가”
> 

Intent가 “왜 묻는가”라면

Cognitive는 “어떻게 보고 싶은가”다.

---

## 4. Cognitive 5개 분류와 PM의 역할

PM은 **각 분류별로 ‘사용할지 말지’와 ‘어떤 조합을 허용할지’**를 결정한다.

---

### 4.1 Dimension – 어디를 나눠서 볼 것인가

> 분해의 기준
> 

| 값 | 의미 |
| --- | --- |
| by_stage | 채용 단계별 |
| by_job | 직무별 |
| by_recruiter | 담당자별 |
| by_source | 지원 경로별 |

### PM 가이드

- “단계별로 보고 싶은 질문인가?”
- “직무별 차이를 보고 싶은가?”

✅ 여러 개 선택 가능

❌ 새로운 분해 축 추가 불가

---

### 4.2 Metric – 무엇을 보고 싶은가

> 지표의 형태 (추상화된 개념)
> 

| 값 | 의미 |
| --- | --- |
| count | 수량 |
| conversion_rate | 전환 비율 |
| dropoff_rate | 탈락 비율 |
| avg_duration | 소요 시간 |

### PM 가이드

- “비율이 중요한 질문인가?”
- “속도/기간이 중요한가?”

❗ 주의

“기술면접 탈락률” 같은 **구체 KPI는 cognitive가 아님**

---

### 4.3 Temporal – 시간을 어떻게 볼 것인가

| 값 | 의미 |
| --- | --- |
| daily | 일 단위 |
| monthly | 월 단위 |
| cohort | 같은 시점 기준 |
| time_series | 추세 |
| future | 미래 |

### PM 가이드

- “이 질문은 특정 시점인가, 흐름인가?”
- “미래를 묻고 있는가?”

---

### 4.4 Comparison – 무엇과 비교할 것인가

| 값 | 의미 |
| --- | --- |
| vs_previous_period | 과거 대비 |
| vs_other_group | 다른 그룹 |
| vs_target | 목표 대비 |

### PM 가이드

- “비교 대상이 있는 질문인가?”
- “없으면 굳이 넣지 말 것”

---

### 4.5 Explanation – 얼마나 깊게 볼 것인가

| 값 | 의미 |
| --- | --- |
| summary | 요약 |
| root_cause | 원인 분석 |

### PM 가이드

- “간단히 보면 되는가?”
- “왜 그런지까지 필요 한가?”

---

## 5. PM이 직접 설계하는 실제 예시

### 예시 1

질문:

> “이번 달 기술면접 탈락률이 왜 이렇게 높아?”
> 

PM 설계:

```json
{
"intent":"diagnosis",
"cognitive":{
"metric":["dropoff_rate"],
"dimension":["by_stage"],
"temporal":["monthly"],
"explanation":["root_cause"]
}
}

```

---

### 예시 2

질문:

> “최근 채용 전환율 추세를 간단히 보고 싶어”
> 

PM 설계:

```json
{
"intent":"analysis",
"cognitive":{
"metric":["conversion_rate"],
"temporal":["time_series"],
"explanation":["summary"]
}
}

```

## 6. PM 자유도에 대한 명확한 경계

### PM이 자유로운 영역

- 질문 문구
- intent 선택
- cognitive 조합
- 어떤 cognitive를 쓰지 않을지 결정

### PM이 건드리면 안 되는 영역

- metric 계산식
- 단계 정의
- 데이터 스키마
- 실행 방식

---

## 7. PM을 위한 체크리스트 (질문 설계 시)

질문 하나당 아래를 체크하면 충분하다.

- [ ]  이 질문의 목적은 하나로 정의되는가?
- [ ]  굳이 나눠서 봐야 하는가?
- [ ]  비율 / 수량 / 시간 중 무엇이 핵심인가?
- [ ]  시간 비교가 필요한가?
- [ ]  요약이면 충분한가, 원인이 필요한가?

---

## 8. 한 문장 요약 (PM용)

> **PM은 “질문이 어떤 의미 구조를 가지는지”를 설계한다.
> 
> 
> 데이터와 계산은 시스템의 책임이다.
> 
> Intent는 목적,
> 
> Cognitive는 관점이다.**
> 

---

이 문서를 기준으로 하면

PM은 **자유롭게 질문을 설계하면서도**,

AI 시스템의 안정성은 **절대 깨지지 않는다**.
