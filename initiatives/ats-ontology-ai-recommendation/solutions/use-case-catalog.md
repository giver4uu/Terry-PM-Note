> **"신뢰 구축이 목표. 복잡하고 정확도 낮은 1개 < 단순하고 정확한 3개"**
>
> MVP는 고객이 "이거 진짜 맞네!"라고 느끼는 순간을 만드는 것이 핵심.
> UC-012/014처럼 데이터 부족 시 정확도가 낮아지는 기능은 신뢰를 깎는다.
> UC-003/011/007처럼 단순하지만 100% 정확한 기능으로 신뢰를 먼저 쌓는다.

### UC-007: 프로세스 병목 진단 (Core MVP)

**Phase:** MVP | **난이도:** 중간 | **RICE 점수:** 66 (v2.0)

#### 스토리

> 스케일업 G사의 HR 매니저 송지현은 "왜 개발자 채용이 3개월이나 걸리죠?"라는 경영진 질문에 명확하게 답하지 못합니다. 온톨로지 기반 병목 진단을 통해 "기술면접 -> 최종면접 단계에서 평균 21일 지연, CTO 일정 조율이 주요 원인"이라는 데이터를 제시하여 프로세스 개선 포인트를 정확히 짚었습니다.

#### 문제 정의 (Job-to-be-Done)

**고객:** HR 매니저, 채용 관리자
**상황:** 채용 프로세스가 왜 오래 걸리는지 원인 파악이 필요할 때
**목표:** 어느 단계에서 얼마나 지연되는지 정량적 데이터로 병목 식별
**문제:** 직감적으로 "느리다"는 건 알지만 정확한 원인 모름

#### 온톨로지 활용

**사용 Objects:**
- Application (current_stage, stage_history)
- Stage Transition (from_stage, to_stage, timestamp, duration)
- Recruitment Stage (stage_name, average_duration, benchmark)
- AI_Recommendation (bottleneck_stage, root_cause, suggested_action)

**사용 Links:**
- PROGRESSES_TO (Stage Transition -> Recruitment Stage)
- FOR_POSITION (Application -> Job Posting)

**쿼리 패턴:**
```
1. 특정 기간/포지션의 모든 Application 조회
2. 각 Application의 Stage Transition 시계열 분석
3. 단계별 평균 체류 시간 계산
4. 벤치마크 대비 이상치 단계 식별
5. 병목 원인 추론 (담당자, 일정, 평가 지연 등)
```

#### AI 철학 준수

| 원칙 | 구현 방법 |
|------|----------|
| **투명성** | "병목 감지: 기술면접 -> 최종면접 평균 21일 (벤치마크 7일 대비 3배). 분석 근거: 최근 3개월 45건 데이터, CTO 관련 17건이 30일+ 지연" |
| **오버라이드** | "이 기간은 예외 (휴가 시즌)" 표시로 분석에서 제외 가능 |
| **학습 루프** | 병목 해소 조치 후 개선 여부 자동 추적, 리포트에 반영 |

#### **실무 검증 (보리) - v2.0 추가**

| 항목 | 내용 |
|------|------|
| **실무 빈도** | 주간 (주 1회 HR 리뷰 미팅 필수 아젠다) |
| **실무자 반응** | **5/5점** "진짜 필요해!" - 경영진 설득 무기 |
| **핵심 우려** | Stage Transition 데이터 품질이 엉망이면 쓰레기 인사이트 |
| **PM 결정** | MVP 유지. 데이터 품질 경고 기능 필수 포함 |

**보리의 코멘트:**
> "ATS 데이터는 실무에서 구멍 많습니다. 담당자가 업데이트 안 하면 AI가 아무리 좋아도 소용없어요. MVP에 데이터 품질 체크 없으면 출시 3개월 후 '이거 안 맞는데요' 클레임 폭주합니다."

#### **데이터 품질 요구사항 - v2.0 추가**

| 항목 | 기준 | 대응 |
|------|------|------|
| **최소 요구** | Stage Transition **70%+** 완성도 | 미달 시 분석 불가 메시지 |
| **권장 수준** | Stage Transition **90%+** 완성도 | "신뢰도 높음" 배지 표시 |
| **품질 경고 UI** | 데이터 누락 건수 표시 | "Stage 업데이트 누락 15건, 분석 신뢰도 중간" |
| **신뢰도 표시** | 모든 분석 결과에 필수 | 녹색/노란색/빨간색 신호등 |
| **보완 워크플로우** | 클릭 몇 번으로 누락 데이터 입력 | "지금 업데이트" 버튼 |

#### 비즈니스 임팩트 측정 지표

| 지표 | 현재 | 목표 | 측정 방법 |
|------|------|------|----------|
| 병목 식별 정확도 | - | 90% | 사용자 피드백 "맞음/틀림" |
| 평균 리드타임 | 60일 | 45일 | 지원 -> 오퍼 수락 기간 |
| 병목 해소율 | - | 50% | 식별 후 3개월 내 개선 비율 |

#### 구현 가이드

**데이터 요구사항:**
- 필수: Stage Transition 타임스탬프 (70%+ 완성도)
- 권장: 각 단계의 담당자 정보
- 최소: 동일 포지션 유형 20건 이상

**MVP 구현 범위:**
- Stage별 평균/중앙값/표준편차 계산
- 이상치 자동 감지 (평균 + 2 표준편차)
- **데이터 품질 경고 UI (필수)**
- 단순 시각화 (바 차트)

### UC-011: 다음 액션 리마인더 (Core MVP - 최우선)

**Phase:** MVP | **난이도:** 낮음 | **RICE 점수:** 62 (v2.0: +6)

#### 스토리

> 스타트업 K사의 1인 채용 담당 최다인은 10개 포지션, 각 20명의 후보자를 관리합니다. "오늘 누구한테 연락해야 하지? 누구 면접 피드백 받아야 하지?" 온톨로지 기반 다음 액션 리마인더를 통해 매일 아침 "오늘의 할 일: 후보자 A 팔로업 (5일 무응답), B 면접 일정 확정 (48시간 내), C 피드백 요청 (김 부장)"이라는 리스트를 받아 놓치는 일이 없어졌습니다.

#### 문제 정의 (Job-to-be-Done)

**고객:** 리크루터, 채용 담당자
**상황:** 다수의 후보자와 프로세스를 동시에 관리할 때
**목표:** 우선순위가 높은 액션을 놓치지 않고 적시에 실행
**문제:** 멀티태스킹으로 중요한 팔로업을 놓침

#### 온톨로지 활용

**사용 Objects:**
- Application (current_stage, stage_entered_at, assigned_recruiter)
- Communication (last_contact, awaiting_response)
- Interview (scheduled_date, feedback_status)
- Task (type, due_date, priority, status)

**사용 Links:**
- ASSIGNS (Recruiter -> Application)
- SCHEDULES (Interview -> Application)
- COMMUNICATES_WITH (Recruiter <-> Candidate)

**쿼리 패턴:**
```
1. 담당 Recruiter의 모든 active Application 조회
2. 각 Application의 상태별 필요 액션 계산
   - 팔로업 필요: 마지막 연락 후 X일 경과
   - 면접 일정: 48시간 내 확정 필요
   - 피드백 대기: 면접 종료 후 미수신
3. 긴급도별 정렬
4. 매일 아침/저녁 리마인더 발송
```

#### AI 철학 준수

### UC-029: "오퍼 협상 시뮬레이션"

**Phase:** Phase 2 | **난이도:** 중간 | **RICE 점수:** 36

#### 스토리 (보리 시나리오)

> 스케일업 BB사의 리크루터 최유진은 후보자에게서 "한 번 더 올려주세요" 요청을 받았습니다. "올려줘야 할까? 올려줘도 거절할 것 같은데..." 온톨로지 기반 오퍼 협상 시뮬레이션을 통해 "이 시점에 5% 추가 제안 시 수락 확률 20%p 상승 (현재 45% -> 65%). 10% 추가해도 수락 확률 유사 (68%)"라는 분석으로 최적 역제안이 가능해졌습니다.

**문제:** 직감에 의존, 과도하게 올려주거나 협상 결렬
